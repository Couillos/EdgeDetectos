2. BINANCE FUTURES
================================================================================

Base URL : https://fapi.binance.com (USDT-M)
           https://dapi.binance.com (COIN-M)
Documentation : https://developers.binance.com/docs

------------------------------------------------------------------------
2.1 FUNDING RATE
------------------------------------------------------------------------
Endpoint      : GET /fapi/v1/fundingRate
Paramètres    : symbol, startTime, endTime, limit (max 1000)
Granularité   : Par settlement (8h par défaut, 4h ou 2h sur contrats récents)
Depuis 2020   : ✅ OUI — données depuis novembre 2019 (lancement Binance
                Futures). Aucune limite de profondeur historique.
Rate limit    : 500 req / 5 min / IP
WebSocket     : Non pour l'historique. Temps réel via markPrice@1s
Réponse       : symbol, fundingRate, fundingTime, markPrice

Exemple de requête :
  GET https://fapi.binance.com/fapi/v1/fundingRate?symbol=BTCUSDT&startTime=1577836800000&limit=1000

------------------------------------------------------------------------
2.2 OPEN INTEREST
------------------------------------------------------------------------
Endpoint (snapshot)  : GET /fapi/v1/openInterest
Endpoint (historique) : GET /futures/data/openInterestHist
Paramètres (hist.)   : symbol, period, limit (max 500), startTime, endTime
Granularité          : 5m, 15m, 30m, 1h, 2h, 4h, 6h, 12h, 1d
Depuis 2020          : ❌ NON — uniquement le dernier mois ("Only the data
                       of the latest 1 month is available")
Rate limit           : 1000 req / 5 min / IP
Alternative          : Collecter les snapshots en temps réel et stocker,
                       ou utiliser Tardis.dev / CoinGlass

------------------------------------------------------------------------
2.3 TAKER BUY/SELL VOLUME
------------------------------------------------------------------------
Endpoint      : GET /futures/data/takerlongshortRatio
Paramètres    : symbol, period, limit (max 500), startTime, endTime
Granularité   : 5m, 15m, 30m, 1h, 2h, 4h, 6h, 12h, 1d
Depuis 2020   : ❌ NON — uniquement les 30 derniers jours
Rate limit    : 1000 req / 5 min / IP
Réponse       : buySellRatio, buyVol, sellVol, timestamp

Attention : Cet endpoint retourne un ratio + volumes buy/sell agrégés,
mais pas le volume taker brut notional séparé.

Alternative pour données depuis 2020 :
  - Aggregated Trades : GET /fapi/v1/aggTrades (isBuyerMaker permet de
    reconstruire le volume taker buy/sell)
  - Klines : GET /fapi/v1/klines incluent takerBuyBaseAssetVolume et
    takerBuyQuoteAssetVolume, disponibles depuis le lancement du contrat

------------------------------------------------------------------------
2.4 LIQUIDATIONS (LONG / SHORT)
------------------------------------------------------------------------
REST (public)       : ❌ GET /fapi/v1/allForceOrders — DÉPRÉCIÉ depuis
                       avril 2021
REST (compte)       : GET /fapi/v1/forceOrders — nécessite API Key,
                       liquidations de l'utilisateur uniquement, 90 jours max
WebSocket            : <symbol>@forceOrder — stream temps réel par symbole
                      !forceOrder@arr — stream temps réel tous marchés
Granularité          : Événementiel (1 snapshot / 1000ms)
Depuis 2020          : ❌ NON — aucun historique agrégé disponible
Alternative          : CoinGlass, Tardis.dev, Coinalyze

------------------------------------------------------------------------
2.5 LONG/SHORT RATIO
------------------------------------------------------------------------
3 endpoints distincts :

a) GET /futures/data/globalLongShortAccountRatio
   → Ratio de TOUS les comptes traders
b) GET /futures/data/topLongShortPositionRatio
   → Ratio des positions des top 20% traders
c) GET /futures/data/topLongShortAccountRatio
   → Ratio des comptes des top 20% traders

Paramètres    : symbol, period, limit (max 500), startTime, endTime
Granularité   : 5m, 15m, 30m, 1h, 2h, 4h, 6h, 12h, 1d
Depuis 2020   : ❌ NON — uniquement les 30 derniers jours
Rate limit    : 1000 req / 5 min / IP

Réponse (global) : longShortRatio, longAccount, shortAccount, timestamp
Réponse (top)    : longShortRatio, longAccount, shortAccount, timestamp

3. BYBIT
================================================================================

Base URL : https://api.bybit.com
Documentation : https://bybit-exchange.github.io/docs/v5
Version API : V5

------------------------------------------------------------------------
3.1 FUNDING RATE
------------------------------------------------------------------------
Endpoint      : GET /v5/market/funding/history
Paramètres    : category (linear/inverse), symbol, startTime, endTime,
                limit (max 200)
Granularité   : Par settlement (8h par défaut, varie : 1h/2h/4h/8h selon
                le contrat — vérifiable via /v5/market/instruments-info,
                champ fundingIntervalHour)
Depuis 2020   : ✅ OUI — contrats inverse depuis fin 2019, contrats linear
                (USDT) depuis mai 2020
Rate limit    : ~600 req / 5 sec / IP (market data)
WebSocket     : tickers.{symbol} — champs fundingRate, nextFundingTime

------------------------------------------------------------------------
3.2 OPEN INTEREST
------------------------------------------------------------------------
Endpoint      : GET /v5/market/open-interest
Paramètres    : category, symbol, intervalTime, startTime, endTime,
                limit (max 200), cursor
Granularité   : 5min, 15min, 30min, 1h, 4h, 1d
Depuis 2020   : ✅ OUI — depuis le lancement du symbole (mai 2020 pour
                les contrats linear)
Rate limit    : ~600 req / 5 sec / IP
Réponse       : openInterest (OI total), timestamp
WebSocket     : tickers.{symbol} — champ openInterestValue

------------------------------------------------------------------------
3.3 TAKER BUY/SELL VOLUME
------------------------------------------------------------------------
Endpoint      : ❌ AUCUN endpoint REST agrégé dédié
Alternatives  :
  a) GET /v5/market/recent-trade — trades bruts avec champ side (Buy/Sell),
     max 1000 par requête, pas d'historique profond via REST
  b) Téléchargement CSV historique :
     https://www.bybit.com/en/derivative-activity/developer
     — Données de trades complètes avec côté taker
     — Contrats inverse depuis nov 2019, linear depuis mai 2020
  c) Fournisseurs tiers : Tardis.dev, CoinGlass
WebSocket     : publicTrade.{symbol} — flux de trades avec champ S (Buy/Sell)

------------------------------------------------------------------------
3.4 LIQUIDATIONS
------------------------------------------------------------------------
Endpoint      : ❌ AUCUN endpoint REST pour l'historique
WebSocket     : allLiquidation.{symbol} — temps réel (push 500ms)
                Champs : S (Buy=position longue liquidée, Sell=courte),
                         v (taille), p (prix de faillite), T (timestamp)
Depuis 2020   : ❌ NON — temps réel uniquement
Alternative   : Tardis.dev (liquidations depuis le 18 décembre 2020),
                CoinGlass

------------------------------------------------------------------------
3.5 LONG/SHORT RATIO
------------------------------------------------------------------------
Endpoint      : GET /v5/market/account-ratio
Paramètres    : category, symbol, period, startTime, endTime,
                limit (max 500), cursor
Granularité   : 5min, 15min, 30min, 1h, 4h, 1d
Depuis 2020   : ✅ OUI — depuis le 20 juillet 2020
Rate limit    : ~600 req / 5 sec / IP
Réponse       : buyRatio, sellRatio (ratio de comptes en position
                longue/courte, de 0 à 1)

4. OKX
================================================================================

Base URL : https://www.okx.com
Documentation : https://www.okx.com/docs-v5
Version API : V5

------------------------------------------------------------------------
4.1 FUNDING RATE
------------------------------------------------------------------------
Endpoint (actuel)   : GET /api/v5/public/funding-rate
Endpoint (historique): GET /api/v5/public/funding-rate-history
Paramètres (hist.)  : instId, before, after, limit (max 100)
Granularité         : 8h (par settlement, 3 settlements/jour)
Depuis 2020         : ⚠️ PARTIEL — ~3 mois via API REST.
                      Téléchargement OKX : depuis mars 2022.
                      Via Tardis.dev : depuis 2019.
Rate limit          : 20 req / 2s / IP
WebSocket           : wss://ws.okx.com:8443/ws/v5/public — canal funding-rate

------------------------------------------------------------------------
4.2 OPEN INTEREST
------------------------------------------------------------------------
Endpoint (instantané) : GET /api/v5/public/open-interest
Endpoint (historique) : GET /api/v5/rubik/stat/contracts/open-interest-history
Endpoint (OI+volume)  : GET /api/v5/rubik/stat/contracts/open-interest-volume
Paramètres (hist.)    : instId, period, limit (max ~100)
Granularité           : 5m, 15m, 30m, 1H, 4H, 1D
Depuis 2020           : ⚠️ PARTIEL — ~3 mois via API Rubik.
                        Téléchargement OKX : depuis sept 2021.
                        Via Tardis.dev : depuis mars 2019.
Rate limit            : 20 req / 2s / IP
Réponse               : timestamp, OI en contrats, OI en coin, OI en USD
WebSocket             : canal open-interest

------------------------------------------------------------------------
4.3 TAKER BUY/SELL VOLUME
------------------------------------------------------------------------
Endpoint (spot)      : GET /api/v5/rubik/stat/taker-volume
Endpoint (contrat)   : GET /api/v5/rubik/stat/taker-volume-contract
Paramètres (contrat) : instId, period, limit (max ~100)
Granularité (spot)   : 5m, 1H, 1D
Granularité (contrat): 5m, 15m, 30m, 1H, 4H, 1D
Depuis 2020          : ⚠️ PARTIEL — ~3 mois (contrat) ou ~6 mois (spot)
Rate limit           : 20 req / 2s / IP
Réponse (contrat)    : timestamp, takerBuyVolume, takerSellVolume (en contrats)

------------------------------------------------------------------------
4.4 LIQUIDATIONS
------------------------------------------------------------------------
Endpoint      : GET /api/v5/public/liquidation-orders
Paramètres    : instType (SWAP/FUTURES/OPTION), instFamily, state (filled),
                before, after, limit (max 100)
Granularité   : Tick-by-tick (événements individuels)
Depuis 2020   : ❌ NON — 7 jours glissants uniquement
                Via Tardis.dev : depuis décembre 2020
Rate limit    : 20 req / 2s / IP
Réponse       : instFamily, instId, details[{bkPx, posSide (long/short),
                side, sz, ts}]

CONSTAT : C'est la plus forte limitation d'OKX — seulement 7 jours
d'historique pour les liquidations.

------------------------------------------------------------------------
4.5 LONG/SHORT RATIO
------------------------------------------------------------------------
4 endpoints disponibles :
  a) GET /api/v5/rubik/stat/contracts/long-short-account-ratio
     → Ratio par currency (5m, 1H, 1D, ~6 mois)
  b) GET /api/v5/rubik/stat/contracts/long-short-account-ratio-contract
     → Ratio par contrat (5m→1D, ~3 mois)
  c) GET /api/v5/rubik/stat/contracts/long-short-account-ratio-contract-top-trader
     → Top traders comptes (5m→1D, ~3 mois)
  d) GET /api/v5/rubik/stat/contracts/long-short-position-ratio-contract-top-trader
     → Top traders positions (5m→1D, ~3 mois)

Depuis 2020   : ⚠️ PARTIEL — ~3 à 6 mois maximum via API
Rate limit    : 20 req / 2s / IP

Granularité           : Par settlement (8h ou 1h selon le contrat)

------------------------------------------------------------------------
5.2 OPEN INTEREST
------------------------------------------------------------------------
Endpoint      : GET /api/v2/mix/market/open-interest
Paramètres    : symbol, productType
Granularité   : ❌ Snapshot en temps réel uniquement — PAS d'historique
Depuis 2020   : ❌ NON — aucun endpoint historique public

------------------------------------------------------------------------
5.3 TAKER BUY/SELL VOLUME
------------------------------------------------------------------------
Endpoint      : GET /api/v2/mix/market/taker-buy-sell
Paramètres    : symbol, period (5m, 15m, 30m, 1h, 2h, 4h, 6h, 12h, 1D)

Réponse       : buyVolume, sellVolume, ts

5.5 LONG/SHORT RATIO

Granularité           : 5m, 15m, 30m, 1h, 2h, 4h, 6h, 12h, 1D, 1Dutc
Depuis 2020           : ⚠️ PARTIEL — depuis ≈ mi-2022

Réponse (comptes)     : longAccountRatio, shortAccountRatio,

7. DERIBIT
================================================================================

Base URL : https://www.deribit.com/api/v2
Documentation : https://docs.deribit.com
Protocole : JSON-RPC 2.0 (REST + WebSocket)
Instruments : BTC-PERPETUAL, ETH-PERPETUAL, SOL-PERPETUAL, etc.

------------------------------------------------------------------------
7.1 FUNDING RATE
------------------------------------------------------------------------
Endpoint      : public/get_funding_rate_history
Paramètres    : instrument_name, start_timestamp (ms), end_timestamp (ms)
Granularité   : 1h (HORAIRE — la plus fine du marché !)
Depuis 2020   : ✅ OUI — depuis 2018 (lancement du BTC-PERPETUAL)
Rate limit    : ~20 req/s (système de crédits)
Autres        : public/get_funding_chart_data (length: 8h, 24h, 1m)
                public/get_funding_rate_value
Réponse       : timestamp, index_price, prev_index_price, interest_8h,
                interest_1h

NOTE : Deribit est le SEUL exchange à proposer un funding rate avec
granularité horaire. Tous les autres se limitent au settlement (8h).

------------------------------------------------------------------------
7.2 OPEN INTEREST
------------------------------------------------------------------------
Endpoint      : public/get_book_summary_by_instrument
Paramètres    : instrument_name
Granularité   : ❌ Snapshot en temps réel uniquement
Depuis 2020   : ❌ NON — pas d'historique via API publique
Alternative   : Collecter les snapshots via WebSocket ticker ou Tardis.dev
WebSocket     : ticker.{instrument_name}.{interval} — champ open_interest

------------------------------------------------------------------------
7.3 TAKER BUY/SELL VOLUME
------------------------------------------------------------------------
Endpoint      : ❌ AUCUN endpoint agrégé dédié
Alternative   : public/get_last_trades_by_instrument_and_time
                — champ tick_direction (indicateur agressif)
                — agrégation manuelle nécessaire
Depuis 2020   : ⚠️ PARTIEL — via trades bruts

------------------------------------------------------------------------
7.4 LIQUIDATIONS
------------------------------------------------------------------------
Endpoint      : public/get_last_settlements_by_currency
                public/get_last_settlements_by_instrument
Paramètres    : currency (BTC/ETH), type (settlement/delivery/bankruptcy),
                count (max 100), continuation, search_start_timestamp (ms)
Granularité   : Par événement de settlement
Depuis 2020   : ✅ OUI — depuis ≈ 2018
Rate limit    : Système de crédits (standard)
Réponse       : instrument_name, position, mark_price,
                session_profit_loss, timestamp

NOTE : Les événements type=bankruptcy correspondent aux liquidations.
Deribit est l'un des rares exchanges à proposer un historique de
liquidations via API REST publique.

------------------------------------------------------------------------
7.5 LONG/SHORT RATIO
------------------------------------------------------------------------
Endpoint      : ❌ AUCUN endpoint dédié
Depuis 2020   : ❌ NON — Deribit ne publie pas ce ratio
Alternative   : Amberdata (ratio calculé à partir des données Deribit)
